PHP性能优化

PHP性能优化难易层次(从易到难)：PHP语言级的优化<PHP周边的优化<PHP自身的优化

PHP语言级的优化

了解PHP代码运行流程

工具：ab、seige、vld、xhprof

注意事项：

  • 尽可能使用PHP内置变量、常量和函数解决问题
  • PHP内置函数的性能优劣；e.g. isset()与array_key_exists()区别
  • 尽可能少用魔法变量；e.g. __get()
  • 建议尽量不适用错误预制符@，@的实际逻辑：在代码开始前和结束后，增加opcode，忽略报错；
  • 合理使用内存，利用unset()及时释放不使用的内存，PS:unset()出现注销不掉的情况；
  • 尽量少用正则表达式，正则表达式的回溯开销大，利用字符串处理函数实现相同的逻辑；
  • 避免在循环内做运算，因为会重复运算；
  • 减少计算密集型业务，PHP语言特性决定，适合衔接WebServer与后端服务、UI呈现；
  • 务必使用带引号字符串做键值，否则PHP会将没有引号的键值当成常量查找

mysql查询优化

2017-10-17

分析查询速度慢的原因

  • 记录慢查询日志

    分析查询日志时，不建议直接打开查询日志进行分析，比较浪费时间和精力，可以使用pt-query-digest工具进行分析

  • show profile

    set profiling = 1；开启，服务器上执行的所有语句会检测消耗的时间，存到临时表中;
    show profiles,
    show prfiles for query 临时表ID。

  • show status

    show status会返回一些计数器，show global status查看服务器级别的所有计数
    有时根据这些计数，可以猜测出哪些操作代价较高或者消耗时间多

  • show processlist;

  • explain别名：desc

    分析单条sql语句

优化查询过程中的数据访问

  • 访问数据太多导致查询性能下降
  • 确定应用程序是否在检索大量超过需要的数据，太多行？太多列？
  • 确认mysql服务器是否在分析大量不必要的数据行
  • 避免使用如下sql语句
    1.查询不需要的记录，使用Limit解决
    2.多表关联查询返回全部列，指定a.id,a.name…
    3.总是去除全部类,select * 会让优化器无法完成索引覆盖扫描的优化
  • 重复查询相同的数据，采用缓存
  • 是否在扫描额外的几率
    1.使用explain来进行分析，如果发现查询需要扫描大量的数据但只是返回少数的行，可以通过如下技巧优化：使用索引覆盖扫描，把所有用的列都放在索引中，这样存储引擎就不需要会表获取对应行就可以返回结果
  • 改变数据库和表的结构，修改数据表的范式，eg:降低范式，空时间换取时间
  • 重写SQL语句，让优化器以更优的方式执行查询

优化长难得查询语句

  • 一个复杂查询还是多个简单查询

    1.MySQL内部每秒能扫描内存中上百万行数据，相比之下，响应数据给客户端就要慢得多；
    2.使用尽可能少的查询是好的，但是有时将一个大的查询分解为多个小的查询是很有必要的

  • 切分查询
    将一个大的查询分解为多个小的相同查询，如大量删除数据时，防止表锁住；

-分解关联查询
可以将一条关联语句分解成多条SQL来执行，让缓存效率更高，执行单个查询可以减少锁的竞争，在应用层做关联可以更容易对数据库进行拆分。查询效率大幅度提升，较少冗余将记录的查询

优化特定类型的查询语句

  • 优化count()查询
    count(*)会忽略所有的列，直接统计所有列数，因此不要使用count(列名)
    myisam中，没有任何where条件下count(*)非常快
    当有where条件是，myisam的count就不一定比其他引擎快
    可以使用explain查询近似值，用近似值代替count(*)
    增加汇总表
    使用缓存

  • 优化关联查询
    确定on或者using子句的列上有索引
    确保group by和order by 中只有一个表中的列，这样MySQL才有可能使用索引

  • 优化子查询–嵌套查询
    尽可能使用关联查询取代

  • 优化group by和distinct
    这两种查询均可使用索引来优化，是最有效的优化方法
    关联查询中，使用标识列进行分组的效率会更高
    如果不需要order by，进行group by时使用order by null,MySQL就不在进行文件排序
    with rollup 超级聚合，可以把操作挪到应用程序处理

  • 优化limit分页
    limit 偏移量大的时候，查询效率较低
    可以记录上次查询的最大ID，下次查询是直接根据该ID来查询
    如：select id from table limit 0,100 where id > 100;

  • 优化union查询
    union all的效率高于union，可以使用union all查询出数据，在通过应用程序进行过滤。

先定位低效的SQL语句，然后根据SQL语句可能低效的原因排查，先从索引着手，如果索引没有问题，考虑以下几个方面，数据访问的问题，长难查询句的问题还是一些特定类型优化的问题。

php-mysql

2017-10-16

mysql数据类型

  • 整数类型

    TINYINT, SMALLINT, MEDIUMINT, INT, BIGINT
    属性：UNSIGNED（非负）
    长度：INT(11)：不会限制取值范围，只是限制显示范围
    零填充：zerofill

  • 实数类型

    1.FLOAT, DOUBLE, DECIMAL
    2.DECIMAL储蓄比BIGINT大的整数，可以用于存储精确的小数
    3.FLOAT和DOUBLE类型支持使用标准的浮点进行近似计算，效率更高，而DECIMAL略似当做字符串处理，所以存储更大

  • 字符串

    1.VARCHAR, CHAR, TEXT, BLOB
    2.VARCHAR使用1或2个额外字节记录字符串的长度，列长度小于255字节，使用1个字节表示，否则用2个
    3.超出varchar指定长度，会截断，char也一样
    4.char会使用空格进行填充以方便比较，而varchar会缩小到字符串长度
    5.对于经常变更的数据，char比varchar更好，char不容易产生碎片
    6.对于非常短的列，char比varchar在存储空间上更有效率
    7.尽量避免使用BLOB/TEXT，查询会使用临时表，会导致严重的性能开销

  • 枚举
    1.有时可以使用枚举代替常用的字符串类型
    2.把不重复的集合存储成一个预定义的集合
    3.非常紧凑，把列表值压缩到一个或两个字节
    4.内部存储是整数
    5.尽量避免使用数字作为ENUM枚举的常量，易混乱
    6.排序是按照内部存储的整数进行排序
    7.枚举会使表大小大大减小，因为存储整数类型

  • 日期和时间类型
    1.尽量使用TIMESTAMP，比datetime空间效率高
    2.用整数保存时间戳的格式不方便处理
    3.用bigint存储微秒

  • 列属性
    auto_increment, default, not null, zerofill

mysql基础操作

  • 常用操作
    1.MySQL的连接和关闭：mysql -u -p -h -P
    2.其他：\G, \c, \q, \s, \h, \d

mysql存储引擎

  • InnoDB

    1.默认事务型引擎
    2.数据存储在共享表空间，可以通过配置分开
    3.对主键查询高于其他类型
    4.内部做了很多优化，从磁盘读取数据时会自动在内存构建hash索引，插入数据是自动构建插入缓冲区
    5.通过一些机制和工具支持真正的热备份
    6.支持崩溃后的安全恢复
    7.支持行级锁
    8.支持外键

  • MyISAM

    1.mysql5.1之前,myisam是默认
    2.拥有全文索引、压缩、空间函数
    3.不支持事务和行级锁，不支持崩溃后的安全恢复
    4.表存储在两个文件，MYD和MYI
    5.设计简单，某些场景下性能很好，eg.count(*)

  • 其他表引擎

    archive， BlackHole， CSV， memory

mysql锁机制

  • 定义：当多个查询同时进行数据修改的时候，就会产生并发控制的问题，锁就是解决问题。
  • 共享锁和排它锁，即系读锁和写锁
  • 读锁：共享的，不堵塞，多个用户可以同时读同一个资源，互不干扰
  • 写锁：排他的，一个写锁会阻塞其他的写锁和读锁
  • 锁粒度：
    1.表锁，系统性能开销小，会锁定整张表，MyISAM是表锁
    2.行锁，最大程度的支持并发处理，开销大，InnoDB是行锁

mysql事务处理

  • InnoDB支持事务处理
  • 服务器层不管理事务，由下层存储引擎实现，所以在同个事务中，不可使用多种存储引擎
  • 在不支持事务表上执行事务操作，不提醒不报错，无效果

存储过程

  • 通过把处理封装在容易使用的单元中，简化复杂的操作，保证数据一致性
  • 存储过程就是处理有业务逻辑和流程的mysql语句集合
  • 简化多变动的管理

触发器

  • 提供给程序员和数据分析员来保证数据完整性的一种方法，他是与表事件相关的特殊的存储过程
  • 使用场景

    1.可通过数据库中的相关表实现级联更改
    2.实时监控某张表的某个字段的更改而需要做出的相应处理
    3.某些业务编号的生成等
    4.滥用会造成数据库及应用程序维护的困难

变量的作用域和静态变量

1.函数体内使用global或$GLOBAL全局数组引用全局变量
2.静态变量static

1.仅初始化一次
2.初始时需要赋值
3.每次执行函数该值会保留
4.static修饰的变量是局部的，仅在函数内部有效
5.可以记录函数的调佣次数，从而可以在某些条件下终止递归。

函数的参数及参数的引用传递

1.默认情况下，函数的参数通过值传递
2.如果希望允许函数修改它的值，必须通过引用传递参数

函数的返回值及引用返回

1.值通过使用可选的返回语句return返回
2.可以返回包括数组和对象的任意类型
3.返回语句终止函数执行，将控制权交回函数调用处
4.省略return,返回值为NULL，不可返回多个值(可采用数组形式)

  • 从函数返回一个引用，必须在函数声明和指派返回值给一个变量时都使用引用运算符&

外部文件的导入

  • include/require语句包含并运行指定文件
  • 如果给出路径名按照路径来找，否则从include_path中查找，如果Include_path中也没有，则从条用脚本文件所在目录和当前工作目录下寻找
  • 当一个文件被包含时，其中所包含的代码继承了include所在行的变量范围（代码复制过来）
  • 加载过程找不到文件,include警告，require致命错误，
  • require在出错是产生E_COMPLIME_ERROR级别的错误，导致脚本中止，include出错是产生E_WARNING警告，脚本继续运行
